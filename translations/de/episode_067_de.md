[NOVA]: Die Model Context Protocol Working Group veröffentlichte den Release Candidate vom 28. Juli 2026 am 8. Juni 2026 und markiert damit die bedeutendste architektonische Änderung in MCPs Geschichte. Die Kernveränderung ist die Verschiebung von einem zustandsbehafteten Sitzungsmodell zu einem zustandslosen Transport mit optionalen zustandsbehafteten Anwendungen obendrauf. Im vorherigen Modell verwalteten MCP-Clients einen Sitzungsspeicher auf der Serverseite. Das war eine Sticky-Session-Einschränkung, die horizontales Skalieren ohne Session-Affinität am Load Balancer unmöglich machte. Jede Anfrage von einem Client musste zurück an denselben Server geleitet werden, der seinen Sitzungszustand hielt, was bedeutete, dass Sie eine MCP-Server-Flotte nicht hinter einen Standard-Load-Balancer stellen und aufskalieren konnten. Das RC entfernt dies, indem es allen Sitzungskontext in die Anfrage-Payload selbst verschiebt. Jeder MCP-Client kann jetzt zu jeder Server-Replik routen, ohne dass ein Sitzungsspeicher erforderlich ist. Das ist eine grundlegende Änderung, wie das Protokoll in der Produktion operiert, und es ist die Änderung, auf die jeder Operator gewartet hat, der MCP-Server hinter einem Load Balancer betreibt.

[ALLOY]: Server-zu-Client-Anfragen werden umstrukturiert. Zuvor konnte ein Server Daten nur durch den Sampling-Callback an den Client pushen, was ein schmaler, spezialisierter Pfad war, der für einen spezifischen Anwendungsfall entwickelt wurde. Das RC fügt eine allgemeine Server-Request-Primitive zur JSON-RPC-Schicht hinzu und ermöglicht es Servern, Prompts, Ressourcen-Updates oder Tool-Ergebnisse zu pushen, ohne dass der Client pollen muss. Dies ist architektonisch ähnlich wie Server-Sent Events in HTTP funktionieren. Ein Server-initiierter Stream mit Client-seitigem Abonnement. Der Server wartet nicht darauf, dass der Client fragt. Er sendet das Update, wenn er etwas zu senden hat. Für Agents, die auf Echtzeit-Serverzustand angewiesen sind, entfernt dies die Polling-Schleife, die Latenz und Overhead hinzufügte.

[NOVA]: Das Extensions-Framework ist jetzt erstklassig. Neue Fähigkeiten werden als Opt-in-Erweiterungen mit ihrem eigenen versionierten Namespace versendet, stabilisieren in Erweiterungsform und gehen nur dann in die Kernspezifikation über, wenn und wenn sie breite Adoption verdienen. Dies ist eine Governance-Änderung ebenso wie eine technische Änderung. Das bedeutet, dass die MCP-Spezifikation nicht mehr das einzige Fahrzeug für neue Funktionalität ist. Erweiterungen können unabhängig vom Kernspezifikationszyklus versendet werden, was dem Ökosystem einen schnelleren Weg gibt, neue Fähigkeiten zu liefern. Die MCP Apps-Erweiterung, die SEP-1865 ist, ermöglicht es Servern, interaktive HTML-Interfaces zu versenden, die in Sandkasten-iframes mit einer kontrollierten postMessage-Brücke gerendert werden. Dies ist ein Server-gerendertes UI-Pattern, das benutzerdefinierte Prompt-Injection-Benutzeroberflächen in Agent-Tooling ersetzen könnte. Server können eine HTML-Oberfläche erstellen, sie an den Client senden, und der Client rendert sie in einem Sandkasten-iframe. Der Agent kann mit dieser Oberfläche interagieren, ohne dass der Client die Logik der Oberfläche verstehen muss. Die Tasks-Erweiterung graduierte von experimentell zu einer benannten Erweiterung, was bedeutet, dass sie eine stabile Schnittstelle und einen definierten Verhaltensvertrag hat.

[ALLOY]: Drei zuvor experimentelle Features werden unter der neuen Feature-Lebenszyklusrichtlinie, SEP-2577, deprecated. Vollständige JSON Schema 2020-12-Validierung für Tools ist jetzt erforderlich anstatt empfohlen, was bedeutet, dass MCP-Tool-Implementierungen schema-konforme Ausgabe produzieren müssen oder der Client sie ablehnen wird. Die Spezifikation ist jetzt strenger, was die richtige Richtung für ein Protokoll ist, das in Produktionsumgebungen adoptiert wird, wo malformed Tool-Responses Agenten-Sitzungen kaputt machen können.

[NOVA]: Die Operator-Oberflächen, die sich ändern, sind die MCP-Server-Implementierungen. Server-Implementierer müssen ihren Code auf serverseitigen Sitzungszustand prüfen und ihn zum Client oder einem externen Speicher migrieren. Das zustandslose Modell bedeutet, dass der Server keinen pro-Client-Zustand zwischen Anfragen mehr hält. Wenn Ihr Server irgendwelchen In-Memory-Zustand hat, der eine Client-Sitzung repräsentiert, muss dieser Zustand auf die Client-Seite verschoben werden, kodiert in der Anfrage-Payload, oder zu einem externen Speicher wie Redis. MCP-Client-Hosts sollten aktualisieren, um die neue Server-Request-Primitive zu behandeln, was bedeutet, Server-initiierte Pushes zu handhaben, die ohne Client-Poll ankommen. Das Extensions-Framework bedeutet, dass neue Fähigkeiten als Opt-in-Ergänzungen anstatt als Spezifikationsänderungen versendet werden. Beobachten Sie Erweiterungsversionen als führende Indikatoren für neue MCP-Funktionalität. Der Erweiterungs-Namespace ist unabhängig vom Kernprotokoll versioniert, was bedeutet, dass Sie auf eine spezifische Erweiterungsversion pinnen können, ohne das Kernprotokoll zu upgraden.

[ALLOY]: Was Agenten jetzt tun können, was früher unmöglich war: MCP-Server-Infrastruktur horizontal ohne "sticky sessions" zu skalieren, was bedeutet, dass eine standardmäßige, lastverteilte Bereitstellung mit mehreren Server-Replikaten jetzt eine unterstützte Topologie und keine Umgehungslösung mehr ist. Es akzeptiert Text-, Bild- und Videoeingaben mit Textausgabe und ist für Agenten-Workloads konzipiert, die GUI-basierte Umgebungen, CLI-Reasoning und cross-modale Wahrnehmung umfassen. [ALLOY]: Die Hybridarchitektur kombiniert effiziente lineare Attention mit sparse Mixture-of-Experts-Routing für starke Skalierbarkeit bei geringeren aktiven Parameterzahlen. Das sparse MoE-Design bedeutet, dass das Modell pro Token eine Untermenge seiner Gesamtparameter aktiviert, was die Inferenzkosten niedrig hält, während Reasoning auf Frontier-Niveau beibehalten wird. Der lineare Attention-Mechanismus verarbeitet lange Kontextsequenzen effizienter als Standard-Attention, was für ein 1M-Token-Kontextfenster wichtig ist, wo Standard-Attention rechnerisch aufwendig wird. Das 1M-Token-Kontextfenster wird über alle Eingabemodalitäten hinweg geteilt, was bedeutet, dass lange Video-Transkripte und große Codebasen in einem einzigen Kontextfenster koexistieren können. Sie können ein zweistündiges Video-Transkript, eine 500.000 Zeilen umfassende Codebasis und eine natürlichsprachliche Aufgabenbeschreibung in denselben Kontext eingeben, und das Modell führt Reasoning über alle davon aus.

[NOVA]: Die multimodale Pipeline-Vereinheitlichung ist die architektonische Story. Die meisten multimodalen Modelle verwenden einen separaten Vision-Encoder, der Bilder verarbeitet und eine Repräsentation erzeugt, die das Sprachmodell konsumiert. Diese Architektur fügt einen Verarbeitungsschritt und eine Übersetzungsebene zwischen der visuellen Eingabe und dem sprachlichen Reasoning hinzu. Qwen3.7-Plus integriert Vision und Sprache in ein einziges Modell ohne separaten Vision-Encoder, was bedeutet, dass die visuelle Eingabe direkt in das Sprachmodell-Backbone fließt. Das Ergebnis ist ein geringerer Tool-Call-Overhead für Bildverständnisaufgaben und ein einheitlicherer Reasoning-Prozess, der ohne Pipeline-Übergabe zwischen visuellen und textuellen Eingaben wechseln kann.
