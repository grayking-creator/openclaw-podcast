Se a página não puder executar a ação, a ferramenta deve retornar um erro estruturado que o agente possa entender: campo ausente, valor inválido, permissão necessária, confirmação necessária, sessão expirada. [ALLOY]: O Chrome DevTools para agents dá aos agentes de codificação uma faixa de verificação de navegador. Isso é adjacente ao WebMCP, mas resolve um problema diferente. WebMCP é sobre uma página declarando ferramentas. DevTools para agents é sobre permitir que um agente de codificação inspecione a página real que ele acabou de alterar.

[NOVA]: Essa distinção importa. Um agente de codificação pode editar o arquivo correto, passar em um teste unitário e ainda assim enviar uma interface quebrada. A página pode ter um erro de console, uma requisição de rede que falhou, um overlap de layout, um problema de breakpoint mobile, um problema de acessibilidade ou uma regressão do Lighthouse. O código fonte sozinho não prova que a experiência do usuário funciona.

[ALLOY]: O DevTools para agents aponta para um loop onde o agente pode lançar ou anexar a um navegador gerenciado, inspecionar o estado da página renderizada, emular tamanhos de viewport, emular geolocalização, depurar uma sessão ativa do Chrome e executar fluxos de trabalho do Lighthouse. Isso dá ao agente evidências do runtime, não apenas do sistema de arquivos.

[NOVA]: A transição do navegador gerenciado é importante porque o trabalho frontend está cheio de pressupostos ambientais. O servidor de desenvolvimento local, rota, viewport, permissões, cookies, geolocalização, comportamento de rede e console do navegador todos importam. Um agente que trabalha apenas com código frequentemente perde esses. Um agente consciente do navegador pode verificá-los diretamente.

[ALLOY]: Emulação responsiva é um bom exemplo. Uma captura de tela desktop pode parecer perfeita enquanto o layout mobile é inutilizável. Se um agente pode trocar viewports, inspecionar a página e capturar evidências, ele pode pegar problemas antes de um revisor humano abrir o telefone e encontrar controles sobrepostos. Emulação de geolocalização importa para busca local, mapas, delivery, viagem, conformidade e qualquer app onde localização muda o que a UI mostra.

[NOVA]: A automação do Lighthouse adiciona uma superfície de auditoria estruturada. Não é toda a verdade, mas dá sinais repetíveis para performance, acessibilidade, melhores práticas e SEO. Para um agente, sinais repetíveis são úteis porque podem se tornar portões: faça a mudança, execute a auditoria, corrija a regressão óbvia, relate a pontuação e os problemas restantes.

[ALLOY]: A recomendação mais ampla para construtores é que agentes de frontend devem ser responsabilizados por evidências renderizadas. Se a tarefa muda a UI, o agente deve rodar o app, inspecionar a página e verificar pelo menos um viewport relevante. Quanto mais estável a integração com DevTools se torna, menos aceitável é para um agente afirmar que uma mudança na UI está concluída sem olhar a página real.

[NOVA]: O ponto de atenção é a confiabilidade da configuração. A verificação no navegador tem várias partes móveis: uma instância do navegador, um servidor dev, navegação de rotas, permissões, possivelmente estado de autenticação e runtime de auditoria. O valor depende de quão consistentemente os agentes podem entrar nesse loop entre frameworks. Mas a direção está certa: edições de código mais evidência do navegador supera edições de código sozinhas. ...

[ALLOY]: Um gate de build prático para agentes de UI pode ser simples. Se a tarefa muda uma página, o agente deve abrir a rota, capturar o status do console, verificar um viewport desktop e um mobile, e reportar qualquer sobreposição visível, controle quebrado ou requisição de rede que falhou. Se a tarefa afeta performance ou acessibilidade, adicione um run do Lighthouse. Esse não é um processo de QA perfeito, mas é muito melhor do que parar nos testes passando.

[NOVA]: A parte importante é que evidência do navegador muda o comportamento do agente. Quando o modelo vê a página renderizada, ele pode perceber que o label de um botão quebra de forma estranha, um modal é alto demais, um gráfico está em branco, ou um estado de loading nunca limpa. Esses não são preocupações abstratas de qualidade. Eles são a diferença entre uma funcionalidade liberada e uma tela quebrada.

[ALLOY]: Isso também cria um artefato de revisão melhor para humanos. Em vez de dizer que o build de frontend está completo, um agente pode dizer qual rota abriu, qual viewport verificou, o que o Lighthouse encontrou e o que ele corrigiu depois. Isso torna o trabalho mais fácil de confiar e mais fácil de questionar. Para trabalho de UI, a confiança deve vir de comportamento de runtime observado, não de confiança no diff do código fonte.
