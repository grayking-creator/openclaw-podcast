[NOVA]: O projeto "architect-loop" demonstrou uma forma altamente eficiente de gerenciar custos de tokens em sistemas multi-agentes separando os papéis de "revisão" e "construção". Ao combinar um revisor Fable com um construtor Codex, o projeto relata uma redução de 80% nos tokens consumidos pelo modelo revisor. Isso é alcançado controlando cuidadosamente quais informações são passadas ao orquestrador em cada etapa do processo.

[ALLOY]: A arquitetura de runtime é construída no princípio de "contexto seletivo". Em um loop de agente tradicional, você pode enviar toda a base de código ou um diff massivo para o modelo em cada passagem. Isso é incrivelmente caro, especialmente se você estiver usando um modelo de ponta para a revisão. O architect-loop muda isso fazendo o "construtor" (o agente de codificação baseado em terminal Codex) fazer o trabalho pesado de escrever o código, e então enviar apenas um resumo de alto nível e artefatos específicos de volta para o "revisor" (Fable). O Fable nunca olha os arquivos fonte brutos em massa; ele só vê o que precisa ver para tomar a próxima decisão arquitetural.

[NOVA]: Esta é uma otimização brilhante porque reconhece que "raciocínio" e "implementação" são tarefas diferentes com requisitos de recursos diferentes. Você não precisa do seu modelo mais caro para escrever um componente React boilerplate ou um endpoint CRUD simples. Você quer que esse modelo lide com a lógica de alto nível e garanta que a mudança não quebre a arquitetura do sistema. Ao rotear o trabalho "de baixo nível" para um modelo mais barato e manter a janela de contexto do modelo "de alto nível" a menor possível, você pode escalar dramaticamente a complexidade das tarefas que seu agente pode lidar sem estourar seu orçamento.

[ALLOY]: Para desenvolvedores, isso valida um padrão onde o orquestrador nunca mantém a carga útil fonte completa em sua sessão ativa. Em vez disso, você mantém um "resumo de estado" que o orquestrador usa para navegar pelo projeto. Esta é uma forma muito mais escalável de construir agentes para repositórios grandes. Se você está executando um loop de agente que atualmente está queimando milhares de dólares por mês em custos de tokens, implementar uma divisão planejador-construtor assim é a maneira mais imediata de melhorar sua eficiência.

[NOVA]: Há um benefício secundário que frequentemente é perdido em análises focadas em custos. Quando o revisor só vê um resumo estruturado, a mesma estrutura de resumo pode ser armazenada em cache na camada de inferência. Se sua função de sumarização for determinística na mesma entrada, você pode atingir a camada de cache de prompts do fornecedor e pular o custo de prefill do revisor em cada passagem subsequente. É aí que os ahorros reais se acumulam: não apenas "prompts menores", mas "prompts menores que também são cacheáveis". Um planejador que nunca releu fonte bruta acaba parecendo uma sequência de chamadas de continuação baratas, que é um perfil de custo muito diferente de um revisor que reprocessa o diff inteiro a cada iteração.

[ALLOY]: A outra lição arquitetural é o limite entre os dois agentes. O architect-loop traça esse limite em "artefatos de revisão estruturados" — resultados de testes, saída de lint, um resumo de diff e uma lista das próximas três ações. Esse é um contrato muito limpo. Se você está construindo sua própria configuração multi-agente, a tentação é passar a saída completa do construtor para o planejador e deixar o planejador descobrir o que fazer com ela. Esse acoplamento é o que torna os modelos revisores caros. Um planejador que só consome um envelope estruturado e estável será muito mais barato de operar e muito mais fácil de trocar quando você quiser atualizar um dos dois modelos independentemente.

[NOVA]: A figura de redução de 80% é a própria medição do projeto, então a leitura correta é "este é um padrão real e reproduzível no código aberto," não "todo modelo revisor mostrará o mesmo número." O formato das economias depende de quão verboso é o seu builder e quanto do código-fonte ele processa por passagem. Mas a afirmação qualitativa — de que contexto seletivo mais uma fronteira estruturada entre planejador e builder é mais barato do que um único agente fazendo ambos os trabalhos — se sustenta na literatura mais ampla de multi-agentes, e architect-loop é a demonstração pública mais limpa disso neste ciclo. ...
