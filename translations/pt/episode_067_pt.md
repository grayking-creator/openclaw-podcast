[NOVA]: O grupo de trabalho do Model Context Protocol publicou o Release Candidate de 28 de julho de 2026 em 8 de junho de 2026, marcando a mudança arquitetural mais significativa na história do MCP. A mudança central é de um modelo de sessão stateful para um transporte stateless com aplicações stateful opcionais empilhadas em cima. No modelo anterior, clientes MCP mantinham um armazenamento de sessão do lado do servidor. Isso era uma restrição de sticky-session que tornava escalonamento horizontal impossível sem afinidade de sessão no load balancer. Cada requisição de um cliente tinha que ser roteada de volta para o mesmo servidor que mantinha seu estado de sessão, o que significava que você não podia colocar uma frota de servidores MCP atrás de um load balancer padrão e escalar. O RC remove isso movendo todo o contexto de sessão para o payload da requisição em si. Qualquer cliente MCP agora pode rotear para qualquer réplica de servidor sem necessidade de armazenamento de sessão. Essa é uma mudança fundamental em como o protocolo opera em produção, e é a mudança que todo operador rodando servidores MCP atrás de um load balancer estava esperando.

[ALLOY]: As requisições de servidor-para-cliente são reestruturadas. Anteriormente, um servidor podia enviar dados ao cliente apenas através do callback de sampling, que era um caminho estreito e especializado projetado para um caso de uso específico. O RC adiciona uma primitiva de requisição geral de servidor à camada JSON-RPC, habilitando servidores a enviarem prompts, atualizações de recursos, ou resultados de ferramentas sem um poll do cliente. Isso é arquiteturalmente similar a como Server-Sent Events funcionam em HTTP. Um stream iniciado pelo servidor com assinatura do lado do cliente. O servidor não espera o cliente perguntar. Ele envia a atualização quando tem algo para enviar. Para agentes que dependem de estado de servidor em tempo real, isso remove o loop de polling que adicionava latência e overhead.

[NOVA]: A estrutura de Extensões agora é de primeira classe. Novas capacidades são enviadas como extensões opt-in com seu próprio namespace versionado, estabilizam em forma de extensão, e apenas se movem para a especificação central se e quando conquistam adoção ampla. Esta é uma mudança de governança tanto quanto uma mudança técnica. Isso significa que a especificação MCP não é mais o único veículo para novas funcionalidades. Extensões podem ser enviadas independentemente do ciclo de especificação central, o que dá ao ecossistema um caminho mais rápido para entregar novas capacidades. A extensão MCP Apps, que é SEP-1865, permite que servidores enviem interfaces HTML interativas renderizadas em iframes sandboxed com uma ponte postMessage controlada. Este é um padrão de UI renderizado pelo servidor que poderia substituir interfaces de usuário de injeção de prompt personalizado em ferramentas de agente. Servidores podem criar uma interface HTML, enviá-la ao cliente, e o cliente a renderiza em um iframe sandboxed. O agente pode interagir com essa interface sem o cliente precisar entender a lógica da interface. A extensão Tasks se gradua de experimental para uma extensão nomeada, o que significa que tem uma interface estável e um contrato de comportamento definido.
