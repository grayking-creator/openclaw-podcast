[NOVA]: Vamos desacelerar nos detalhes do Hermes por mais alguns minutos, porque o novo release tem vários recursos pequenos que vão importar mais em produção do que em um scan rápido do changelog. Worker heartbeat é o óbvio. Um heartbeat transforma silêncio em um sinal medido. Se o worker está vivo, o board pode continuar esperando. Se o heartbeat para, o sistema pode marcar a tarefa como stale, fazer reclaim ou mostrar um blocked state. Esse é um primitive básico de confiabilidade, e é exatamente o tipo de primitive que produtos de agente precisam. ...

[ALLOY]: Stale reclaim é o recurso companheiro. Sem reclaim, um worker que falha pode deixar uma tarefa parecendo owned para sempre. Com reclaim, o board pode decidir que a ownership expirou. O detalhe importante é que o reclaim não deve fingir que a tentativa anterior teve sucesso. Deve preservar o failure mode, retry count, evidência e último estado conhecido. Construtores devem inspecionar esse comportamento depois da atualização. Uma tarefa reclamada deve ser auditable, não magicamente fresh. ...

[NOVA]: Zombie detection no Darwin soa como uma nota de rodapé de plataforma, mas é outro exemplo da mesma direção de produto. Runs de agente locais frequentemente dependem de child processes: shell commands, browser processes, tool servers, audio generation, image generation ou test runners. Se um processo morre de forma ruim, a camada de agente precisa saber. Detectar zombie state não é uma copy de marketing empolgante; é como um produto de automação local para de confundir "process exists" com "work is healthy". ...

[ALLOY]: Checkpoint pruning merece uma leitura similar. Persistência que nunca faz prune eventualmente se torna sua própria fonte de outage. Disco enche. Resume fica lento. Estado antigo compete com estado atual. Um feature de checkpoint só é maduro quando inclui limits, cleanup e failure behavior claro. Depois do Hermes v2026.5.7, o teste inteligente é criar vários runs, interromper um, retomar um e inspecionar o diretório de checkpoint ou entrada de banco de dados quanto a size, freshness e recoverability. ...

[NOVA]: A mudança de provider plugin também é maior do que parece à primeira vista. Um plugin boundary permite que essa integração se torne mais explícita. Também dá aos operadores um lugar para testar auth errors, request timeouts, fallback behavior, token limits e formatos de resposta específicos de provedor sem reescrever o resto do produto. ...

[ALLOY]: Trabalho de reparo do MCP está na mesma categoria. Servidores Model Context Protocol podem expor file search, issue search, browser actions, app APIs ou serviços internos. O comando é simples, mas a implicação do produto é grande: trabalho de agente local está se tornando inspectable. ...

[ALLOY]: O campo hook terminalSequence é igualmente prático. Feedback de terminal não é só decoração. Um hook pode precisar tornar uma blocked action visível imediatamente, mudar um título de terminal, tocar uma campainha ou sinalizar completion enquanto o usuário está em outro pane. Output estruturado de hook dá às equipes uma forma mais limpa de conectar policy a feedback. O teste importante é se hooks permanecem predictable em sessões interativas, sessões em background e blocked tool calls. ...

[NOVA]: Plugin source control importa por razões de supply chain. Se uma ferramenta de codificação pode carregar plugins, então source policy é parte da segurança. HTTPS cloning, managed settings, approved sources e workspace identity não são glamorosos, mas reduzem ambiguidade sobre de onde veio o código de extensão executável. Equipes devem tratar configuração de plugin como configuração de dependência: known source, known version, known permission e known update path. ...

[ALLOY]: Workload identity scoping com ANTHROPIC_WORKSPACE_ID aponta na mesma direção. Um CLI local pode parecer pessoal, mas muitos usuários agora o executam dentro de projetos de empresa, managed environments ou dev boxes compartilhados. Identidade precisa seguir o boundary do projeto e boundary da policy. Se o workspace ID errado vazar para o projeto errado, billing, audit e permission checks podem todos se tornar messy. A nova configuração merece um teste de upgrade pequeno, mas deliberado. ...

[NOVA]: O artigo do ADK do Google é importante porque faz uma distinção limpa entre model context e application state. Model context é o texto que o modelo vê. Application state é o registro durável que o app possui. Agentes de longa execução precisam de ambos, mas não devem confundir um pelo outro. O exemplo current_step do artigo é simples de propósito. Um pequeno valor de estado pode prevenir uma grande classe de erros quando o agente é retomado depois de um delay. ...

[ALLOY]: O detalhe do DatabaseSessionService também diz aos construtores para onde o Google acha que essa categoria está indo. SQLite local para desenvolvimento, Cloud SQL para deployment mais sério e writes de estado através de tool calls: esse não é um padrão de exemplo. É uma arquitetura de aplicação normal com uma camada de IA adicionada. O agente ainda pode raciocinar, escrever e chamar ferramentas, mas o app possui o registro durável. Isso torna debugging e compliance mais fáceis porque o estado pode ser inspecionado fora do modelo. ...

[NOVA]: Webhook-driven resume é o momento em que o recurso para de ser teórico. Imagine uma aprovação chega doze horas depois. A mesma ideia se aplica a support cases, onboarding, aprovações financeiras, device provisioning e qualquer tarefa onde o tempo passa entre request e response. ...

[ALLOY]: Avaliações douradas para estado atrasado são especialmente úteis. Times frequentemente testam apenas a primeira requisição e a resposta final. O bug geralmente está no meio: após um timeout, após um resultado parcial de ferramenta, após o usuário mudar uma resposta, ou após um evento externo chegar duas vezes. Pré-configurar o estado permite que uma avaliação comece exatamente nessa condição do meio. Isso transforma um cenário extremo assustador do mundo real em um caso de teste repetível. É uma das ideias mais valiosas no post do ADK. ...

[NOVA]: A API de tarefas do Copilot do GitHub é importante por um motivo diferente: ela faz o código de agentes se encaixar em uma plataforma de repositório. O endpoint pode criar uma tarefa, anexá-la a um repo, selecionar um modelo, escolher uma base ref, e opcionalmente criar um pull request. Isso permite que outro serviço trate o trabalho do Copilot como um job assíncrono. O resultado pode ser observado através de status, branch e artefatos de PR em vez de desaparecer em um chat de assistente. ...

[ALLOY]: O modelo de permissões é a primeira coisa para estudar na documentação. Permissões refinadas de repositório reduzem a necessidade de tokens amplos. Um serviço que inicia tarefas deve ter um token restrito, um conjunto conhecido de repositórios, uma política clara de base-ref, e um requisito de revisão antes que mudanças pousem. ...

[NOVA]: Os estados das tarefas são tão importantes quanto o endpoint de início. Na fila significa aceito, não terminado. Em progresso significa ativo, não seguro. Inativo pode significar que o agente está esperando ou precisa de atenção. Aguardando usuário é um estado do produto, não um erro genérico. Concluído significa que um artefato existe, não que o artefato está correto. Falhou significa que o chamador precisa de um caminho de erro que diga a um humano o que aconteceu. Esses estados dão aos times estrutura suficiente para construir monitoramento em torno de código de agentes. ...

[ALLOY]: A criação de pull-request é o limite de artefato sensato. O novo recurso deve tornar o código mais rastreável, não menos. Se um time adota tarefas do Copilot e ignora a revisão, ele perdeu a melhor parte da atualização.
