[NOVA]: CodeGraphContext es un tipo diferente de herramienta de contexto. Donde agentmemory almacena memorias, CodeGraphContext indexa la estructura del código en una base de datos de grafo y expone ese grafo tanto a humanos como a agentes.

[ALLOY]: El diseño de modo dual es la idea clave. El mismo índice de grafo sirve dos superficies: una CLI donde un desarrollador puede ejecutar consultas en lenguaje natural contra las bases de código, y un servidor MCP donde agentes de IA como Claude Code, Codex, Gemini y Cursor pueden llamar herramientas que responden preguntas estructurales. La CLI te da el mismo grafo para exploración manual. El MCP le da a tu agente el mismo grafo como una herramienta invocable.

[NOVA]: El motor de indexación analiza código usando nodos tree-sitter y construye un grafo de componentes: funciones, clases, métodos, parámetros, relaciones de herencia, llamadas de funciones e importaciones. Eso significa que el grafo conoce la diferencia entre una función con el mismo nombre en la clase A y una función con el mismo nombre en la clase B. Una búsqueda de texto devolvería ambas. El grafo devuelve solo la que se está consultando.

[ALLOY]: La superficie MCP es donde aparece el flujo de trabajo del agente. Las herramientas expuestas incluyen get_callers, get_callees, get_class_hierarchy y get_call_chain. Un agente que necesita entender el radio de explosión de una refactorización puede preguntar al grafo por llamadores y llamados directamente en lugar de grepear las bases de código e intentar razonar sobre lo que encontró. En una base de código grande, la diferencia es significativa. La consulta de grafo es una sola llamada de herramienta con una respuesta precisa, la ruta de grep es una cadena de inferencias inciertas.

[NOVA]: La observación de archivos en vivo actualiza el grafo en tiempo real a medida que los archivos cambian, así el índice no se vuelve obsoleto durante una sesión de refactorización. Los idiomas soportados cubren veintidós, incluyendo Python, JavaScript, TypeScript, Java, C, C++, C#, Go, Rust, Ruby, PHP, Swift, Kotlin, Dart y varios otros. La amplitud del soporte de idiomas es una fortaleza real, y la base de tree-sitter es lo que hace práctico el cobertura de idiomas en lugar de una afirmación de marketing.

[ALLOY]: El backend de base de datos de grafo es conectable. El predeterminado es FalkorDB Lite. La elección del backend importa para las características de rendimiento y el ajuste operativo, y el diseño conectable significa que un equipo puede elegir el backend que coincida con su infraestructura de base de datos de grafo existente.

[NOVA]: El caso de uso práctico es un agente que necesita hacer refactorización no trivial o análisis de código. Con un grafo, el agente puede hacer preguntas precisas y obtener respuestas precisas. El grafo reduce la sobrecarga de llamadas de herramientas en bases de código más grandes donde una búsqueda de texto tendría que ejecutarse en docenas de archivos, y elimina los falsos positivos que vienen de funciones con el mismo nombre en diferentes clases.

[NOVA]: RepoBrain es la tercera herramienta de contexto, y el planteo es que no todos los equipos necesitan el conjunto completo de características de agentmemory. Algunos equipos solo necesitan un registro de decisiones liviano.

[ALLOY]: El diseño es CLI-first y local. Sin infraestructura externa, sin base de datos de grafos, sin grafo de conocimiento. Solo una forma estructurada de almacenar decisiones y patrones que un agente puede consultar en una sesión posterior. El objetivo es el recall de decisiones y almacenamiento de patrones. Cuando un agente toma una decisión sobre cómo resolver un problema o estructurar código, esa decisión se almacena para que una sesión futura pueda consultarla y evitar volver a debatir el mismo punto.

[NOVA]: El enfoque liviano es el punto. agentmemory es un sistema completo con puntuación de confianza, gestión del ciclo de vida y un grafo de conocimiento. CodeGraphContext es una base de datos de grafos con una CLI y superficie MCP. RepoBrain es una herramienta CLI. El tradeoff es profundidad de funcionalidades contra simplicidad operacional, y para un desarrollador que solo quiere que un agente recuerde "decidimos usar el patrón strategy aquí, no herencia," el camino liviano es el ajuste correcto.

[ALLOY]: El cableado específico para agentes cubre Claude Code y Codex para integración drop-in. El registro de decisiones es la estructura de datos clave. El camino de recall es una consulta contra el registro. La ganancia en eficiencia del context-window es la propuesta de valor: en lugar de re-derivar una decisión cada sesión, el agente lee la decisión almacenada y sigue adelante. En proyectos de larga duración, eso es un ahorro real de tiempo.

[NOVA]: La evaluación honesta es que RepoBrain ocupa un nicho diferente a las otras dos herramientas. No está tratando de ser un servidor de memoria compartido entre agentes, y no está tratando de ser una base de datos de grafos. Está tratando de ser la herramienta más pequeña posible que resuelve el problema del recall de decisiones. Para equipos que aún no han adoptado una herramienta de memoria y quieren un punto de entrada de baja fricción, RepoBrain vale la pena考虑. Para equipos que ya necesitan memoria compartida entre agentes o análisis de código respaldada por grafos, agentmemory y CodeGraphContext son las opciones más capaces.

[ALLOY]: Las tres herramientas juntas describen una categoría en maduración. agentmemory es la opción broad, cross-agent. CodeGraphContext es la opción estructural. RepoBrain es la opción liviana.
