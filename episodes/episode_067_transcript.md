## [02:00] OpenAI Codex rust-v0.138.0: CLI-to-desktop handoff, image path exposure, and reasoning effort flexibility

[NOVA]: OpenAI shipped Codex rust-v0.138.0 on June 8, 2026 as a stable release, available from the GitHub releases page under the rust-v0.138.0 tag. This is the first stable release since rust-v0.137.0 appeared in EP065, and it lands with three user-facing features that directly change how builders work with Codex. The headline addition is the slash app command and its companion desktop handoff mechanism. On macOS and native Windows, running slash app in a Codex CLI session transfers the current thread context into Codex Desktop, authenticated by the existing session token with no re-prompt. Windows workspace launches can now open directly into Desktop instead of stopping at a manual prompt, which means a builder can start a task in a terminal, realize they need a GUI browser or multi-window context, and hand off without losing state. The transport uses a named-pipe bus over the local workstation. The desktop client picks up the session token from the CLI's auth store, which means the handoff is authenticated by the existing session rather than requiring a fresh login.

[ALLOY]: The second feature is local image path exposure. When Codex generates an image or attaches a local image to the conversation, the model now receives the saved file path as a first-class reference rather than an ephemeral URL. Previously, follow-up edits required the model to track a URL that could expire or become inaccessible. Now the model holds the exact file reference and can reference it in subsequent tool calls reliably. This fixes the image-follow-up-edit workflow that broke in every release since image generation shipped. The technical mechanism is straightforward: the local file URI is surfaced at the materialize boundary before the model processes the tool result, so Codex receives the path directly rather than holding a separate image URL. The practical impact is that image generation pipelines no longer need URL-tracking logic. The model holds the file path across turns.

[NOVA]: The third feature is reasoning effort flexibility. The TUI now has fallback shortcuts for terminals that miss Alt key bindings, which is a real problem on non-US keyboard layouts where the Alt modifier does not behave as the TUI expects. The model-defined effort levels now flow through in the order the model advertises them rather than being re-ranked by the client, which makes effort-level selection deterministic across Codex deployments. Previously, a developer on a German or French keyboard layout had no reliable way to select reasoning effort because the Alt-based shortcuts did not fire. The fallback shortcuts bypass the Alt binding requirement entirely.

[ALLOY]: Two bug fixes address real operational pain. Clipboard race conditions in high-latency environments were a problem when Codex runs over remote SSH or high-network-jitter connections, where the clipboard write and the model read could happen in the wrong order and corrupt the context. Intermittent authentication failures during long-running sessions were a session token expiry edge case that triggered re-auth prompts mid-task. Neither fix is dramatic on its own, but both were user-reported in the OpenAI developer forum, and both are exactly the class of failure that erodes trust in agent tooling. Operators who run Codex over remote connections or in long-running background sessions will notice the difference immediately.

[NOVA]: The surfaces that changed are the slash app command in the CLI, image path handling in the model interface layer, and reasoning effort fallback shortcuts in the TUI. There are no new config flags or environment variables. Migration risk is low. This is an additive release with backward-compatible defaults. What agents can now do that was previously impossible or brittle: hand off a CLI session to a desktop client mid-task without losing context; reliably reference a generated image file in follow-up edits across turns; select reasoning effort levels on non-US keyboard layouts without workarounds. The named-pipe transport over the local workstation bus is the architecture that makes the handoff reliable. It is not a cloud sync or a file-based state transfer. It is a direct pipe from the CLI process to the desktop client, secured by the existing session token, which means the context transfer is instantaneous and authenticated. [PAUSE]

## [09:00] MCP July 2026 Release Candidate: stateless protocol, server-initiated requests, and the extensions framework

[NOVA]: The Model Context Protocol working group published the July 28, 2026 Release Candidate on June 8, 2026, marking the most significant architectural change in MCP's history. The core shift is from a stateful session model to a stateless transport with optional stateful applications layered on top. In the previous model, MCP clients maintained a session store on the server side. That was a sticky-session constraint that made horizontal scaling impossible without session affinity at the load balancer. Every request from a client had to route back to the same server that held its session state, which meant you could not put an MCP server fleet behind a standard load balancer and scale out. The RC removes this by moving all session context into the request payload itself. Any MCP client can now route to any server replica with no session store required. That is a fundamental change in how the protocol operates in production, and it is the change that every operator running MCP servers behind a load balancer has been waiting for.

[ALLOY]: Server-to-client requests are restructured. Previously, a server could push data to the client only through the sampling callback, which was a narrow, specialized path designed for one specific use case. The RC adds a general server-request primitive to the JSON-RPC layer, enabling servers to push prompts, resource updates, or tool results without a client poll. This is architecturally similar to how Server-Sent Events work in HTTP. A server-initiated stream with client-side subscription. The server does not wait for the client to ask. It sends the update when it has something to send. For agents that rely on real-time server state, this removes the polling loop that added latency and overhead.

[NOVA]: The Extensions framework is now first-class. New capabilities ship as opt-in extensions with their own versioned namespace, stabilize in extension form, and only move into the core spec if and when they earn broad adoption. This is a governance change as much as a technical change. It means the MCP spec is no longer the only vehicle for new functionality. Extensions can ship independently of the core spec cycle, which gives the ecosystem a faster path to deliver new capabilities. The MCP Apps extension, which is SEP-1865, lets servers ship interactive HTML interfaces rendered in sandboxed iframes with a controlled postMessage bridge. This is a server-rendered UI pattern that could replace custom prompt-injection user interfaces in agent tooling. Servers can author an HTML interface, ship it to the client, and the client renders it in a sandboxed iframe. The agent can interact with that interface without the client needing to understand the interface's logic. The Tasks extension graduates from experimental to a named extension, which means it has a stable interface and a defined behavior contract.

[ALLOY]: Three previously experimental features are deprecated under the new feature lifecycle policy, which is SEP-2577. Full JSON Schema 2020-12 validation for tools is now required rather than recommended, which means MCP tool implementations need to produce schema-compliant output or the client will reject them. The spec is stricter now, which is the right direction for a protocol that is being adopted in production environments where malformed tool responses can break agent sessions.

[NOVA]: The operator surfaces that change are the MCP server implementations. Server implementors need to audit their code for server-side session state and migrate it to the client or an external store. The stateless model means the server no longer holds per-client state between requests. If your server has any in-memory state that represents a client session, that state must move to the client side, encoded in the request payload, or to an external store like Redis. MCP client hosts should update to handle the new server-request primitive, which means handling server-initiated pushes that arrive without a client poll. The Extensions framework means new capabilities will ship as opt-in additions rather than spec changes. Watch extension versions as leading indicators of new MCP functionality. The extension namespace is versioned independently of the core spec, which means you can pin to a specific extension version without upgrading the core protocol.

[ALLOY]: What agents can now do that was previously impossible: scale MCP server infrastructure horizontally without sticky sessions, which means a standard load-balanced deployment with multiple server replicas is now a supported topology rather than a workaround. Receive server-initiated pushes without polling, which means agents can respond to server state changes in real time rather than waiting for the next polling interval. Use server-rendered UI surfaces as interactive components within agent sessions, which opens a new class of human-in-the-loop workflows where the server provides a structured interface that the agent navigates. ## [20:00] Qwen3.7-Plus: multimodal agent model, hybrid MoE architecture, and GUI/CLI reasoning

[NOVA]: Qwen3.7-Plus is the balanced multimodal variant of the Qwen 3.7 generation, available via API through Alibaba Cloud Model Studio and listed on OpenRouter. It accepts text, image, and video input with text output, and it is designed for agent workloads that span GUI-based environments, CLI reasoning, and cross-modal perception. Early third-party testing covered browser-based workflows, OpenCode coding tasks, C++ game generation, frontend design, flight simulation, and interactive website generation. The model handles multimodal inputs natively without a separate vision encoder pipeline, which reduces tool-call overhead for image-understanding tasks. This is architecturally significant. A single model handles vision and language reasoning, which means agents can process a screenshot, a code file, and a natural-language instruction in the same context window without routing through separate vision and language models.

[ALLOY]: The hybrid architecture combines efficient linear attention with sparse mixture-of-experts routing for strong scalability at lower active parameter counts. The sparse MoE design means the model activates a subset of its total parameters per token, which keeps inference costs down while maintaining frontier-level reasoning. The linear attention mechanism handles long-context sequences more efficiently than standard attention, which is important for a 1M-token context window where standard attention becomes computationally expensive. The 1M-token context window is shared across all input modalities, which means long video transcripts and large codebases can coexist in a single context window. You can feed a two-hour video transcript, a 500,000-line codebase, and a natural-language task description into the same context and the model reasons across all of them.

[NOVA]: The multimodal pipeline unification is the architectural story. Most multimodal models use a separate vision encoder that processes images and produces a representation that the language model consumes. That architecture adds a processing step and a translation layer between the visual input and the language reasoning. Qwen3.7-Plus integrates vision and language into a single model without a separate vision encoder, which means the visual input flows directly into the language model backbone. The result is lower tool-call overhead for image-understanding tasks and a more unified reasoning process that can switch between visual and textual inputs without a pipeline handoff.

[ALLOY]: The operator surfaces are the API via Alibaba Cloud Model Studio and OpenRouter. No open-weight release is available. The hybrid MoE architecture means Qwen3.7-Plus can run efficiently on commodity GPU budgets while maintaining frontier-level reasoning, but the API delivery model means operators are not running it locally. For agents that need to understand screens, parse documents, or automate GUI-based workflows, Qwen3.7-Plus provides a single unified model that handles images and video alongside text without routing through separate vision and language models. Test it as a drop-in replacement for Claude Opus 4.7 on tasks that require visual understanding combined with long-context code reasoning.

[NOVA]: What agents can now do that was previously impossible: handle multimodal agent tasks with a single unified model instead of a pipeline of separate vision and language models, which simplifies the agent tooling configuration and reduces the latency from image input to language reasoning. Process long video transcripts and large codebases in a single context window, which enables a new class of agent tasks that combine visual understanding with deep code reasoning.
