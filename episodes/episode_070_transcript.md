[ALLOY]: A detailed technical post on imil.net has captured the attention of the local LLM community, documenting a heterogeneous dual-GPU configuration that achieves enterprise-grade throughput on consumer hardware. The setup pairs an RTX 5080 with an RTX 3090 to run Qwen 3.6 27B at 8-bit quantization, sustaining over 80 tokens per second. This is a significant milestone because it demonstrates that builders don't need a unified high-VRAM accelerator like an A100 or H100 to achieve high-speed inference on models in the 30-billion-parameter class.

[NOVA]: The technical mechanism here is a clean tensor-parallel layer split. In this configuration, the transformer layers of the Qwen model are distributed across the available VRAM of both cards. Because the 5080 and 3090 combined offer a substantial memory pool, the entire Q8 quantized payload fits comfortably with enough headroom for a large KV cache. The inference runs on a backend similar to llama.cpp, where the runtime assigns a contiguous block of layers to each GPU. This avoids the need to swap weights in and out of system memory, which is the primary killer of local inference performance.

[ALLOY]: The bottleneck in a setup like this is typically the PCIe host bandwidth. Unlike enterprise setups that use NVLink or Infinity Fabric for high-speed inter-GPU communication, these consumer cards have to send activation tensors across the motherboard's PCIe bus. Every time the model finish processing the layers on the first card, the resulting data must be transferred to the second card to continue the pass. However, at 8-bit quantization, the size of those activation tensors is small enough that the PCIe overhead doesn't prevent the system from hitting that 80-token-per-second mark. For context, most human readers consume text at about 5 to 10 tokens per second, so this setup provides an experience that is effectively instantaneous for interactive use.

[NOVA]: Builders are looking at this as a blueprint for repurposing older hardware. The RTX 3090 remains a favorite in the community because of its 24 gigabytes of VRAM, and pairing it with a newer, faster card like the 5080 allows the system to benefit from the newer card's compute power while utilizing the older card's capacity. The Q8 quantization is the key trade-off; while it significantly reduces the memory footprint and increases speed, there is a measurable, though often slight, impact on reasoning quality compared to the full 16-bit precision model. For agentic workflows where speed and cost-per-token are the primary constraints, this is a trade-off many are willing to make.

[ALLOY]: The broader implication is the continued viability of the "mismatched" rig. We are seeing more sophisticated logic in model runners to handle these heterogeneous pools. The observation is that as long as you can fit the model across the combined VRAM, the specific generation of the silicon matters less than the total memory bandwidth and the PCIe version of the slot it's sitting in. [ALLOY]: The runtime architecture is built on the principle of "selective context." In a traditional agent loop, you might send the entire codebase or a massive diff to the model on every pass. This is incredibly expensive, especially if you are using a high-end model for the review. Architect-loop changes this by having the "builder" (the terminal-based coding agent Codex) do the heavy lifting of writing the code, and then only sending a high-level summary and specific artifacts back to the "reviewer" (Fable). Fable never looks at the raw source files in bulk; it only sees what it needs to see to make the next architectural decision.

[NOVA]: This is a brilliant optimization because it acknowledges that "reasoning" and "implementation" are different tasks with different resource requirements. You don't need your most expensive model to write a boilerplate React component or a simple CRUD endpoint. You want that model to handle the high-level logic and ensure the change doesn't break the system's architecture. By routing the "low-level" work to a cheaper model and keeping the "high-level" model's context window as small as possible, you can dramatically scale the complexity of the tasks your agent can handle without blowing your budget.

[ALLOY]: For builders, this validates a pattern where the orchestrator never holds the full source payload in its active session. Instead, you maintain a "state summary" that the orchestrator uses to navigate the project. This is a much more scalable way to build agents for large repositories. If you are running an agent loop that is currently burning thousands of dollars a month in token costs, implementing a planner-builder split like this is the most immediate way to improve your efficiency.

[NOVA]: There is a secondary benefit that often gets missed in cost-focused writeups. When the reviewer only ever sees a structured summary, the same summary structure can be cached at the inference layer. If your summarization function is deterministic on the same input, you can hit the provider's prompt-cache layer and skip the reviewer's prefill cost on every subsequent pass. That is where the real savings stack up: not just "smaller prompts," but "smaller prompts that are also cacheable." A planner that never re-reads raw source ends up looking like a sequence of cheap continuation calls, which is a very different cost profile from a reviewer that re-processes the whole diff on every iteration.

[ALLOY]: The other architectural lesson is the boundary between the two agents. Architect-loop draws that boundary at "structured review artifacts" — test results, lint output, a diff summary, and a list of the next three actions. That is a very clean contract. If you are building your own multi-agent setup, the temptation is to pass the builder's full output through to the planner and let the planner figure out what to do with it. That coupling is what makes reviewer models expensive. A planner that only ever consumes a stable, structured envelope will be much cheaper to operate and much easier to swap when you want to upgrade one of the two models independently.

[NOVA]: The 80% reduction figure is the project's own measurement, so the right read is "this is a real and reproducible pattern in the open codebase," not "every reviewer model will show the same number."
